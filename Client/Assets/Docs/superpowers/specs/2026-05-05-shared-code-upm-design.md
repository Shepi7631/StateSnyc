# Shared Code via UPM Local Package

Date: 2026-05-05

## Problem

`Share/Pathfinding.Core/` is the source-of-truth for pathfinding algorithms. Currently `Client/Assets/Pathfinding.Core/` holds a manual copy that has already diverged (missing files, differing comments). Server does not reference it at all. Goal: both Client and Server reference the single Share source directly, with zero manual sync.

## Chosen Approach

Unity Local Package (UPM) with `file:` protocol for Client; `ProjectReference` for Server.

## Architecture

### Directory Layout (after)

```
Share/Pathfinding.Core/
├── package.json                    (UPM package descriptor)
├── Pathfinding.Core.asmdef         (Unity Assembly Definition)
├── Pathfinding.Core.csproj         (dotnet build / test, unchanged)
├── Algorithm/
│   ├── AStarSolver.cs
│   ├── FunnelSolver.cs
│   └── NavMeshBuilder.cs
└── Data/
    ├── NavMesh2D.cs
    ├── NavTriangle.cs
    ├── Polygon.cs
    ├── Portal.cs
    ├── Vec2.cs
    └── NavMeshAsset.cs
```

### Client Integration

1. `Client/Packages/manifest.json` gains:
   ```json
   "com.statesync.pathfinding-core": "file:../../Share/Pathfinding.Core"
   ```
2. Delete `Client/Assets/Pathfinding.Core/` (the stale copy).
3. `Client/Assets/Plugins/` must contain:
   - `Triangle.dll` (already present)
   - `System.Text.Json.dll` (add for NavMeshAsset support)
4. Existing scripts (`NavMeshDemoController`, etc.) continue to `using Pathfinding.Algorithm` — no code changes needed since namespace stays the same.

### Server Integration

`Server/StateSync.Server/StateSync.Server.csproj` gains:
```xml
<ProjectReference Include="..\..\Share\Pathfinding.Core\Pathfinding.Core.csproj" />
```

No other changes. Server already targets net9.0 and can consume netstandard2.1 libraries.

### Share/Pathfinding.Core — New Files

#### package.json

```json
{
  "name": "com.statesync.pathfinding-core",
  "version": "1.0.0",
  "displayName": "Pathfinding Core",
  "description": "A* and Funnel algorithms for NavMesh pathfinding",
  "unity": "2021.3"
}
```

#### Pathfinding.Core.asmdef

```json
{
  "name": "Pathfinding.Core",
  "rootNamespace": "Pathfinding",
  "references": [],
  "includePlatforms": [],
  "excludePlatforms": [],
  "allowUnsafeCode": false,
  "overrideReferences": true,
  "precompiledReferences": [
    "Triangle.dll",
    "System.Text.Json.dll"
  ],
  "autoReferenced": true,
  "defineConstraints": [],
  "noEngineReferences": true
}
```

Key settings:
- `overrideReferences: true` + `precompiledReferences`: tells Unity to link against specific Plugin dlls.
- `noEngineReferences: true`: this is pure algorithm code, no UnityEngine dependency.
- `autoReferenced: true`: all Assembly-CSharp scripts can use it without explicit asmdef reference.

### .meta Files

Unity will generate `.meta` files inside `Share/Pathfinding.Core/` for every file and folder. These must be committed to git — Unity uses them for stable GUIDs and import settings. They do not affect `dotnet build` (only `.cs` files are compiled by the csproj).

### Dependency Handling

| Library | Share (dotnet) | Client (Unity) |
|---------|---------------|----------------|
| Triangle.NET | NuGet PackageReference | `Client/Assets/Plugins/Triangle.dll` |
| System.Text.Json | NuGet PackageReference | `Client/Assets/Plugins/System.Text.Json.dll` |

Both resolve at compile time through their respective mechanisms. No conditional compilation needed.

### Assembly-CSharp.csproj

This file is auto-generated by Unity. After the package is added and old copy deleted, Unity will regenerate it on next domain reload. The new version will reference `Pathfinding.Core.dll` (from the asmdef) instead of compiling the pathfinding sources inline.

## What Does NOT Change

- Namespaces: `Pathfinding.Algorithm`, `Pathfinding.Data` — unchanged.
- Consumer code: all `using` statements remain the same.
- Share unit tests: `Pathfinding.Core.Tests` still references the csproj via ProjectReference.
- Proto/network code: unaffected, orthogonal concern.

## Verification Criteria

1. `dotnet test Share/Pathfinding.Core.Tests/` passes (Share still builds independently).
2. Unity opens without errors; `NavMeshDemoController` runs in Play Mode.
3. Server builds with `dotnet build Server/StateSync.Server/`.
4. Editing a file in `Share/Pathfinding.Core/` triggers Unity recompile automatically.
5. No duplicate source files remain in `Client/Assets/Pathfinding.Core/`.
